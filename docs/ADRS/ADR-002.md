# ADR-002: Redis-Based Rate Limiting Strategy

* **Status:** Accepted
* **Date:** 2025-01-01

## Context
Phase 1 introduces multi-tenant quotas and requires enforcing both per-key and per-IP throttles across multiple gateway replicas.

## Decision
Use Redis (shared cache) to implement a sliding window counter. Keys are namespaced as `rate:<scope>:<identifier>:<window>` with TTL equal to the window length. The limiter is applied twice per request: once for the API key and once for the client IP address.

## Consequences
* Rate limit checks remain O(1) and horizontally scalable because each gateway replica increments the same Redis key.
* TTL automatically clears counters, avoiding manual cleanup.
* Redis already required for result caching, minimizing additional infrastructure.
* Future enhancements (per-tenant quotas, burst handling) can extend the key schema without schema migrations.
