# ADR-001: Service Split and gRPC Interface

* **Status:** Accepted
* **Date:** 2025-01-01

## Context
The original monolithic Flask app bundled UI rendering, evaluation, and rate limiting in a single process. Phase 1 requires isolating the evaluation logic for security and scalability while enabling async request handling.

## Decision
We split the system into two deployable units:
- `gateway` FastAPI service handling HTTP/WebSocket requests, authentication, rate limiting, caching, and audit logging.
- `safe-evaluator` microservice exposing a gRPC interface (`/evaluator.v1.Evaluator/Evaluate`) that executes expressions inside a sandbox.

A gRPC-style transport was chosen to provide a strongly-typed, low-latency boundary that can evolve towards mTLS-secured communication in later phases. For offline development environments we ship a lightweight asyncio transport compatible with gRPC semantics.

## Consequences
* Gateway can scale independently from the evaluator worker pool and future job orchestrators.
* The API boundary is explicit and testable with contract tests.
* Additional services (symbolic engine, job orchestrator) can reuse the evaluator RPC without linking directly to sandbox internals.
