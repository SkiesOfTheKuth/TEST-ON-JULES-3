CALCULATOR PLATFORM - PHASE 2 USAGE GUIDE
========================================

Overview
--------
The calculator now operates as a production-ready, distributed platform:
- **Symbolic Engine**  FastAPI service with `/v1/symbolic` endpoint, Redis-backed cache, Prometheus metrics, and subprocess sandbox for SymPy workloads.
- **Gateway** – FastAPI service with synchronous `/calculate` and asynchronous `/jobs` endpoints, per-tenant policy engine, WebSocket notifications, and Celery orchestration.
- **Safe Evaluator** – gRPC microservice performing sandboxed evaluations with strict allowlists, complexity budgets, and telemetry.
- **Workers** – Standard, heavy, and GPU Celery lanes consuming Redis queues and publishing live status updates.
- **Infrastructure** – Postgres (API keys, jobs, policies), Redis (broker, cache, policy store), Grafana/Prometheus/Tempo/Loki for full observability.

Quick Start (Docker Compose)
---------------------------
1. Ensure Docker Desktop/Engine is running.
2. From the repository root execute:
   ```bash
   make compose-phase2-up
   ```
3. Default endpoints:
   - Gateway API: <http://localhost:8080>
   - Grafana: <http://localhost:3000> (`admin` / `grafana`)
   - Postgres: `localhost:5432` (`calculator` / `calculator`)
   - Redis: `localhost:6379`
4. Seed an API key (idempotent):
   ```bash
   poetry -C services/gateway run python -m app.scripts.seed_api_key --api-key local-dev-key --owner "Local Dev" --force
   ```
5. Submit a synchronous calculation:
   ```bash
   curl -X POST http://localhost:8080/calculate \
     -H "X-Api-Key: local-dev-key" \
     -H "Content-Type: application/json" \
     -d '{"expression": "add(4, multiply(3, 2))"}'
   ```
6. Tear everything down (including volumes):
   ```bash
   make compose-phase2-down
   ```

Asynchronous Jobs
-----------------
1. With the stack running, enqueue a standard job:
   ```bash
   curl -X POST http://localhost:8080/jobs \
     -H "X-Api-Key: local-dev-key" \
     -H "Content-Type: application/json" \
     -d '{
           "input_expression": "pow(2, 12) + sqrt(16)",
           "context": {"scale": 3},
           "priority": 1,
           "tags": ["demo"]
         }'
   ```
   The response includes `id`, queue assignment, and WebSocket link.
2. Route to heavy math by adding a `heavy` tag or long-running expression:
   ```bash
   curl -X POST http://localhost:8080/jobs \
     -H "X-Api-Key: local-dev-key" \
     -H "Content-Type: application/json" \
     -d '{
           "input_expression": "integrate(sin(x) * exp(-x**2), x)",
           "tags": ["heavy"]
         }'
   ```
3. Hint GPU lane with a `gpu` tag (requires tenant policy allowing GPU):
   ```bash
   curl -X POST http://localhost:8080/jobs \
     -H "X-Api-Key: local-dev-key" \
     -H "Content-Type: application/json" \
     -d '{
           "input_expression": "sin(x) + cos(x)",
           "tags": ["gpu"],
           "context": {"x": 1.1}
         }'
   ```
4. Poll for completion:
   ```bash
   curl -H "X-Api-Key: local-dev-key" http://localhost:8080/jobs/<job-id>
   ```
5. Stream status updates via WebSocket (requires `npm i -g wscat`):
   ```bash
   wscat -c ws://localhost:8080/ws/jobs/<job-id> -H "X-Api-Key: local-dev-key"
   ```

Symbolic Solve API
------------------
1. Evaluate a symbolic expression directly via the gateway:
   ```bash
   curl -X POST http://localhost:8080/v1/symbolic/solve \
     -H "X-Api-Key: local-dev-key" \
     -H "Content-Type: application/json" \
     -d '{
           "expr": "sin(x)**2 + cos(x)**2",
           "subs": {"x": 1.0}
         }'
   ```
   Sample response:
   ```json
   {
     "ok": true,
     "result": {
       "simplified": "1",
       "latex": "1",
       "evaluated": 1.0,
       "cached": false
     },
     "metadata": {"mode": "symbolic", "cache": false}
   }
   ```
2. Dispatch a symbolic job (routes through the same service but records history):
   ```bash
   curl -X POST http://localhost:8080/jobs \
     -H "X-Api-Key: local-dev-key" \
     -H "Content-Type: application/json" \
     -d '{
           "input_expression": "sin(x)**2 + cos(x)**2",
           "context": {"x": 1.0},
           "mode": "symbolic"
         }'
   ```
   Example WebSocket payload once cached/succeeded:
   ```json
   {
     "id": "<job-id>",
     "status": "succeeded",
     "result_payload": {
       "result": {
         "simplified": "1",
         "latex": "1",
         "evaluated": 1.0,
         "cached": false
       },
       "metadata": {"mode": "symbolic", "cache": false}
     }
   }
   ```

Integration Testing
-------------------
- Spin up the minimal symbolic stack for unit tests:
  ```bash
  docker compose -f docker-compose.test.yml up -d redis symbolic gateway
  pytest -q
  docker compose -f docker-compose.test.yml down -v
  ```
- Bring up the full validation stack and run integration tests:
  ```bash
  make integration
  ```
  or, with services already running:
  ```bash
  poetry -C services/gateway run pytest tests/integration -m integration
  ```
- Coverage highlights:
  - Queue routing across standard, heavy, and GPU lanes.
  - Tenant policy allow/deny flows and cache invalidation.
  - WebSocket lifecycle streaming.
  - Failure propagation (sandbox validation errors) and metrics assertions.
  - Autoscaling decision helper sanity checks.

Load Testing
------------
- Locust script (`tests/load/locustfile.py`) drives job submission/polling with enforced thresholds:
  ```bash
  API_KEY=local-dev-key make load-test
  ```
- Tunable environment variables (defaults in parentheses):
  - `LOCUST_MAX_P95_MS` (`750`) – failure if p95 latency exceeds this value.
  - `LOCUST_MIN_RPS` (`5`) – failure if average throughput drops below this rate.
  - `LOCUST_MAX_FAILURE_RATIO` (`0.05`) – failure if error ratio exceeds this cap.

Observability
-------------
- Grafana: `http://localhost:3000` (admin / grafana) hosts **Gateway Overview**, **Phase 2 Queue Lanes**, and **Worker Health** dashboards covering request health, per-lane depth/autoscale events, and worker utilisation/failure trends.
- Metrics: `curl -s http://localhost:8080/metrics` to inspect counters (`calculator_gateway_jobs_enqueued_total`, `calculator_gateway_policy_violations_total`, etc.) and gauges (`calculator_gateway_job_queue_depth`, `calculator_gateway_worker_cpu_percent`).
- Traces: Tempo spans include `calculator.queue`, `calculator.tenant`, and `calculator.policy_outcome`; filter by `job.id` to debug latency or policy decisions.
- Logs: `docker compose -f docker-compose.phase2.yml logs gateway` streams structured JSON (request IDs, queue decisions, tenant metadata).
- Runbooks: see `docs/OPERATIONS.md` for Celery worker lifecycle, Redis outage recovery, policy tuning, and dashboard interpretation guidance.

Autoscaling & SLOs
------------------
- Dry-run scale decisions from the repo root:
  ```bash
  python scripts/autoscale_workers.py --queue-depth 90 --active-workers 4 --p95-wait 6 --worker-cpu 82
  ```
- Add `--apply` to grow/shrink Celery pools directly (Celery must be reachable); use `--stdin` to feed JSON snapshots exported from Prometheus.
- Default thresholds (`services/gateway/app/config.py > AutoscaleSettings`): scale up at >=75% queue depth, >5s p95 wait, or >85% CPU; scale down once backlog is <=15% and latency/CPU halve; cooldown 180s with 30s drain timeout.
- Grafana `Phase 2 Queue Lanes` dashboard visualises per-lane depth, throughput, and `calculator_gateway_autoscale_events_total`.

Observability

Troubleshooting
---------------
- **403 policy_violation** – verify tenant policy (`tenant_policies` table) permits the requested queue/feature and clear the Redis cache key `policy:<tenant>`.
- **503 queue full** – adjust `GATEWAY_JOB__MAX_QUEUE_SIZE`or scale workers; observe `calculator_gateway_job_queue_depth` before re-enqueueing.
- **Evaluator errors** – check safe evaluator logs, ensure allowlist and complexity thresholds cover the expression, or adjust policy `max_runtime_ms` for long-running jobs.
