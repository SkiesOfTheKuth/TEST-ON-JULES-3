CALCULATOR PLATFORM – PHASE 1 USAGE GUIDE
=============================================

Overview
--------
The calculator now runs as a hardened, multi-service platform:
- `gateway` FastAPI service handling REST/WebSocket entry points, rate limiting, and auditing.
- `safe-evaluator` sandboxed microservice running expressions via a subprocess guard.
- Shared infrastructure: Redis for rate limiting/cache, Postgres for API keys and audit logs, and an observability stack (Prometheus, Tempo, Loki, Grafana).

Quick Start with Docker Compose
-------------------------------
1. Install Docker Desktop (or Docker Engine) and ensure it is running.
2. Copy `.env.development` to `.env` if you want to override defaults.
3. From the repository root run:
   ````bash
   make compose-up
   ````
4. The following ports will be exposed by default:
   - Gateway API: http://localhost:8080
   - Grafana: http://localhost:3000 (admin / grafana)
   - Postgres: localhost:5432 (calculator / calculator)
5. Seed an API key by opening a shell in the gateway container and running the Alembic seeding script (see `OPERATIONS.md`).
6. Send a request with the API key header:
   ````bash
   curl -H "X-Api-Key: <your-key>"         -H "Content-Type: application/json"         -d '{"expression": "add(1, 2)"}'         http://localhost:8080/calculate-sync
   ````
7. To stop the stack run:
   ````bash
   make compose-down
   ````

Local Development (without Docker)
----------------------------------
1. Create a virtual environment and install dependencies:
   ````bash
   python -m venv .venv
   source .venv/bin/activate
   make bootstrap
   ````
2. Start Postgres and Redis locally (Docker compose is the easiest way).
3. Apply database migrations:
   ````bash
   cd services/gateway
   alembic upgrade head
   ````
4. Run the evaluator service:
   ````bash
   make run-evaluator
   ````
5. In another terminal run the gateway:
   ````bash
   make run-gateway
   ````

Asynchronous Jobs
-----------------
Phase 2 adds an async job API backed by Celery and Redis. Use it for heavyweight expressions or batch workloads.

1. Submit a job:
   ````bash
   curl -fsS -X POST http://localhost:8080/jobs \
     -H "X-Api-Key: <your-key>" \
     -H "Content-Type: application/json" \
     -d '{
           "input_expression": "heavy_compute(100000)",
           "context": {"precision": 8},
           "priority": 1,
           "tags": ["demo", "retryable"]
         }'
   ````
   The response returns the job metadata, including polling links and WebSocket URI (`/ws/jobs/<id>`).
2. Poll for completion:
   ````bash
   curl -fsS -H "X-Api-Key: <your-key>" http://localhost:8080/jobs/<job-id>
   ````
   Successful jobs return the cached payload; failures include the error string recorded in Postgres and Prometheus (`jobs_failed{queue,task}`).
3. Optional: stream updates over WebSockets (requires `wscat` or similar):
   ````bash
   npx wscat -H "X-Api-Key: <your-key>" -c ws://localhost:8080/ws/jobs/<job-id>
   ````
4. Explore metrics and dashboards:
   - `curl -s http://localhost:8080/metrics | grep jobs_`
   - Grafana → `calculator-phase2` dashboard → Async panels for throughput, success rate, wait time, and WebSocket health.

Testing
-------
Execute unit tests from the repository root:
````bash
make test
````

API Security Highlights
-----------------------
- All requests require an `X-Api-Key` header referencing a hashed record in Postgres.
- Redis-based sliding window throttles requests per key and per client IP.
- Expressions are validated via AST inspection before hitting the sandbox.
- Successful evaluations may be cached for deterministic expressions.
